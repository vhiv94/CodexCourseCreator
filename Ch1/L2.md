# L2: Build a Simple Banner

## Concept

In Python, **text in quotes** is a **string literal**—characters your code can print or treat as one chunk. You can **concatenate** (join) literals with `+` into a longer string before printing, or you can use several **`print`** calls with separate strings. Each `print` normally ends with a **line break**, so multiple `print` calls usually show up as **multiple lines** in the terminal.

Lesson 1 required **exactly one** line of output. Here you build a **banner**: a **small block of several lines** that introduces your program (a text “title area”). You still **do not** read from the keyboard—only what the script **prints** when it runs matters for the tests.

How the tests count lines: they take everything your program printed, turn Windows-style line breaks into the usual `\n` form, remove one layer of trailing newlines at the end, then split on newline characters. Every line that remains must contain **visible characters**—not a line that is empty or only spaces/tabs. (That matches `line.strip() != ""` in `Ch1/L2.py`.)

## Why it matters in this project

Your course project is a **command-line study log**. Real CLI tools greet the user with a short labeled area—app name, purpose, maybe a hint—before deeper behavior appears. Building a simple multi-line banner now practices the same string and printing ideas you will reuse for prompts, labels, and formatted log output later.

## Common misconceptions

- **“A blank line looks nice between sections.”** — A line that is empty—or only spaces—fails this lesson’s tests. Use visible characters on every line (for example a row of dashes with text) if you want separation.

- **“More lines is always better.”** — The tests cap the banner at **20** lines. Aim for a **short** banner that fits the spirit of the spine’s `project_step`.

- **“This is still a one-line welcome like L1.”** — L2 **requires multiple lines** (at least two). If you only print one line, the multi-line test will fail.
