# Normalize Event Streams with map/filter/reduce

This lesson moves from single-record correctness to stream-level correctness. A functional pipeline treats event processing as a sequence of deterministic transforms: map to normalize each row, filter to remove events that should not enter state, and reduce to fold validated events into an operational output collection. The tools are `map`, `filter`, `reduce` (or equivalent pure transformations), plus immutability-minded handling so input lists are not mutated as a side effect.

This stage matters because every later stock, queue, and ledger operation depends on stream hygiene. If invalid events leak through, bugs appear far away from their origin. A key tradeoff is readability versus strict functional purity; you can compose pure steps cleanly, but still need naming and structure that teammates can debug. Common misconception: pipelines are only about style. Here, they enforce deterministic contracts and keep behavior testable. Reflection prompt: where should filtering of no-op events happen so downstream classes remain simpler?
