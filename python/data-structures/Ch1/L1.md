# L1: Represent Documents as a Sequenced Corpus

## Concept

An **ordered sequence** is a collection where **position matters**: document *A* before document *B* stays a meaningful statement about how you want to traverse the corpus later. Python‚Äôs **`list`** is the usual ordered container‚Äî**appends** go at the end, indexing `0..n-1` is stable until you change the list yourself.

For this course, each **document** is a pair of ideas: a **numeric id** (your handle into the corpus) and the **text body** you will eventually index and search. The spine asks for **stable ids**: the order you receive pairs in is the **logical order** the corpus keeps, and every id must be **unique** so ‚Äúget me document 7‚Äù is unambiguous.

```python
# Illustration: an ordered list of (id, text) pairs‚Äîshape only, not a full solution.
corpus = [(10, "alpha"), (2, "bravo")]
```

## Why it matters in this project

The in-memory search stack treats the corpus as the **ground truth** for what exists and in what order. Later lessons walk this sequence on purpose, slice text, and scan bodies. If ids collide or the list aliases mutable caller state in dangerous ways, indexing and retrieval tests become flaky or wrong. L1 locks in a small, explicit **corpus module** with **deterministic** build and lookup behavior.

## Common misconceptions

- **‚ÄúI can use a dict keyed by id for the corpus.‚Äù** ‚Äî The first behavioral test requires **`build_corpus`** to return a **`list`** that mirrors order; a bare dict alone is the wrong shape unless you still satisfy the **list** contract the test asserts.
- **`ValueError` vs `KeyError`** ‚Äî **Duplicate at build** ‚Üí **`ValueError`** in the tests; **Lookup of an id that was never loaded** ‚Üí **`KeyError`**.
- **Shallow copy vs alias** ‚Äî Appending to the **same list object** the learner passed in must not grow the corpus after `build_corpus` returns; reassignment `rows[i] = ...` must not retroactively swap stored bodies for still-valid ids.
