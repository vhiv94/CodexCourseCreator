# Model Inventory Events with Typed Records

In this lesson, you define a canonical inventory event record so every downstream component speaks the same data language. The core idea is invariant-driven modeling: regardless of how messy raw input is, normalized output should always expose the same fields and predictable value shapes. The immediate tools are dictionary field validation, string normalization (`strip`, case normalization), and stable key naming so consumers can rely on fixed contracts instead of defensive guessing.

This matters because the rest of the control tower assumes normalized events are trustworthy before indexing, queueing, or replaying anything. If record shape drifts now, every later class and function accumulates brittle edge handling. Common misconceptions are that event shape can be \"mostly\" consistent or that key casing is cosmetic; both usually produce silent downstream bugs. Reflection prompt: if two services disagree on whether `kind` is uppercase or lowercase, which component should absorb that inconsistency and why?

## Assigned task(s)

- **Build/Change:** Implement `normalize_inventory_event(raw_event)` in `main.py` so one canonical event structure is returned for valid input.
- **Expected outcomes / acceptance criteria:**

  - [ ] Numeric movement values are preserved as movement deltas, and the function behaves deterministically for repeated identical input.
