# Build Allocation Maps for Open Orders

This lesson formalizes how demand is mapped to available supply. Allocation mapping is a multi-index dictionary problem: orders reference SKUs and quantities, while stock is partitioned by location. Your goal is to build deterministic matching outputs that downstream wave planning can consume without reinterpreting intent. The tools are nested mappings, controlled quantity accounting, and explicit representation of allocated slices.

This matters because poor allocation logic causes hidden under-fulfillment or over-allocation long before dispatch begins. A key tradeoff is local simplicity versus global correctness: greedy local fills can be easy to write but need careful accounting to ensure requested quantity is exactly represented. Common misconception: allocation is complete once any candidate location exists. In practice, quantity completeness and reproducibility are the contract. Reflection prompt: why should total allocated quantity per order line be the first invariant you verify?

## Assigned task(s)

- **Build/Change:** Implement `build_allocation_map(orders, stock)` in `main.py` for deterministic order-to-stock allocation outputs.
- **Expected outcomes / acceptance criteria:**
  - [ ] Output contains allocation entries for each eligible order line.
  - [ ] Allocated quantity equals requested quantity when stock is sufficient.
  - [ ] Allocation entries reference valid stock locations for the corresponding SKU.
  - [ ] Results are deterministic for repeated identical order and stock inputs.
- **Evidence (tests):** `main_test.py` + `lesson_ch4_l1`
