# Build a Functional KPI Reporting Pipeline

This lesson turns operational state into decision-ready metrics through pure reporting transforms. You will generate KPI outputs from snapshot inputs without mutating source state, using map/filter/reduce-friendly composition. The tools are pure function design, ratio calculations, targeted filtering (for example stockout detection), and deterministic report schemas.

In the project arc, this is the observability layer that tells whether the control tower is healthy. The tradeoff is between compact one-liners and transparent calculations that teammates can validate under pressure. Common misconception: reporting is separate from system correctness. If KPI logic mutates input or drifts in schema, downstream dashboards become unreliable. Reflection prompt: which KPI would you trust most when triaging a suspected fulfillment regression and why?

## Assigned task(s)

- **Build/Change:** Implement `build_kpi_report(snapshot)` in `main.py` as a pure reporting transform.
- **Expected outcomes / acceptance criteria:**
  - [ ] Input snapshot is not mutated by report generation.
  - [ ] Report includes fill-rate style ratio metrics computed from snapshot values.
  - [ ] Report includes pending-restock counts and stockout SKU signals.
  - [ ] Output remains deterministic for repeated identical snapshots.
- **Evidence (tests):** `main_test.py` + `lesson_ch6_l2`
