# Compose an InventoryControlTower Service

This lesson creates a single orchestration boundary for all previously built structures and flows. Instead of isolated utilities, you now coordinate normalized event ingestion, stock updates, replenishment queueing, and audit tracking through one service object. The key tools are constructor dependency wiring, batch-processing contracts, and snapshot generation for external consumers.

This integration step matters because production systems are judged at composition points, not isolated helper functions. The tradeoff is between convenience and explicit architecture: a monolithic method can hide coupling, while explicit coordination interfaces remain testable and maintainable. Common misconception: integration code is just glue and needs less rigor. In reality, it is where cross-component invariants live. Reflection prompt: which invariant should a snapshot guarantee even if internal data structures evolve?

## Assigned task(s)

- **Build/Change:** Implement `InventoryControlTower` in `main.py` with batch processing and state snapshot capabilities.
- **Expected outcomes / acceptance criteria:**
  - [ ] Processing a batch reports how many events were handled.
  - [ ] Snapshot output exposes coherent stock state for affected SKUs/locations.
  - [ ] Snapshot includes pending replenishment signals when reorder conditions are reached.
  - [ ] Integration behavior is deterministic for repeated identical batches.
- **Evidence (tests):** `main_test.py` + `lesson_ch6_l1`
- **Hints:** This lesson has optional guidance in `Ch6/L1-hint.md`.
