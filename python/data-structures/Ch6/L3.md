# Simulate Failure Recovery with Undo and Replay

This capstone lesson tests operational resilience, not just happy-path throughput. You model a failure boundary, then verify the system can recover using previously built stack and queue semantics plus replay logic. The tools are deterministic failure injection, replay accounting, and explicit recovery outcome reporting.

This matters because real systems fail in the middle of work, and confidence depends on reproducible recovery behavior. The tradeoff is between aggressive retry speed and state safety: fast recovery paths can cause duplication if replay contracts are unclear. Common misconception: if recovery \"usually\" works, it is good enough. Recovery correctness requires explicit, testable guarantees. Reflection prompt: which recovery metric best tells you that state reconciliation was complete rather than partial?

## Assigned task(s)

- **Build/Change:** Implement `simulate_failure_recovery(events, fail_after)` in `main.py` for deterministic fault simulation and replay reporting.
- **Expected outcomes / acceptance criteria:**
  - [ ] Function reports successful recovery status for tested scenarios.
  - [ ] Function reports how many events were processed in final recovered state.
  - [ ] Function reports replay activity with coherent counts.
  - [ ] Recovery outcome is deterministic for repeated identical inputs.
- **Evidence (tests):** `main_test.py` + `lesson_ch6_l3`
