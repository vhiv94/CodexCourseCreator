# Insert and Remove Ledger Entries Safely

This lesson focuses on mutation safety under linked-list invariants. Removing or inserting entries is not just value editing; it is structural surgery where pointer updates, head/tail maintenance, and length consistency must all stay valid together. The tools are predicate-driven removal logic, defensive mutation sequencing, and invariant checks after each operation.

This matters because ledger corruption propagates silently into reporting, replay, and recovery flows. The core tradeoff is speed of mutation versus correctness guarantees: shortcut pointer updates can appear to work on happy paths but fail under boundary cases (middle removal, near-tail updates, empty/one-node transitions). Common misconception: if traversal still prints values, structure is fine. Broken metadata often appears only later. Reflection prompt: what minimal invariant checks would you run after each mutation during manual debugging?

## Assigned task(s)

- **Build/Change:** Extend `MovementLedgerLinkedList` in `main.py` with safe mutation behavior for removing matched entries while preserving invariants.
- **Expected outcomes / acceptance criteria:**
  - [ ] Removing the first matching entry returns the removed payload.
  - [ ] Remaining entries preserve original relative order.
  - [ ] `head`, `tail`, and length stay coherent after removal.
  - [ ] Boundary cases (removing from middle or near tail) do not corrupt traversal.
