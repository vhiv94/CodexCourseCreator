# Functional Aggregations over Linked Ledgers

With structural correctness in place, this lesson turns ledger traversal into business summaries. You apply map/reduce-style thinking to custom object collections so aggregates remain deterministic and testable even when the underlying container is class-based. The tools are pure summary functions, iteration over ledger-compatible rows, and per-SKU accumulation logic.

This step matters because operational decisions depend on trustworthy aggregate signals, not just raw events. A tradeoff appears between one-pass compact reducers and more explicit multi-step calculations: concise code can be elegant but harder to audit under pressure. Common misconception: functional aggregation only belongs to built-in lists. If your custom class yields stable rows, pure aggregation still works well. Reflection prompt: which aggregate would you trust first when reconciling a suspected stock discrepancy?

## Assigned task(s)

- **Build/Change:** Implement `summarize_movements(rows)` in `main.py` to produce inventory movement summaries suitable for operational checks.
