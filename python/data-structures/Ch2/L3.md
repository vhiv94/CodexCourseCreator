# Compose Stack and Queue in an Intake Service

This lesson shifts from single data-structure correctness to service-level composition. You will coordinate `AuditStack`, `ReplenishmentQueue`, and stock-index state through one intake boundary so incoming events trigger the right side effects in the right order. The key tools are object composition, dependency injection of prebuilt structures, and explicit event processing outcomes (for example, flags indicating whether a restock task was enqueued).

Composition matters here because later chapters assume orchestration behavior is centralized and deterministic. A major tradeoff is convenience versus clarity: tightly coupled classes can be quick to write but hard to validate, while explicit dependencies make testing and future change easier. Common misconception: integration code does not need strict contracts because components are already tested. In practice, composition bugs are often where production issues begin. Reflection prompt: when processing a pick event near reorder threshold, which state transitions must be visible immediately and which can be deferred?

## Assigned task(s)

- **Build/Change:** Implement `IntakeCoordinator` in `main.py` to route processed inventory events into audit and replenishment workflows while updating stock state.
- **Expected outcomes / acceptance criteria:**
  - [ ] Processing a qualifying event returns a structured result that includes whether restock work was enqueued.
  - [ ] Relevant processed events are pushed to the audit stack.
  - [ ] Reorder-threshold behavior queues the expected SKU when stock drops to trigger conditions.
  - [ ] Coordinator behavior stays deterministic for repeated identical event sequences.
