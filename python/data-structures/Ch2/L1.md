# Build an Audit Stack Class

This lesson introduces a custom class for LIFO behavior so rollback-ready audit operations are explicit and testable. A stack is an abstract data type with precise semantics: the most recent event is the first one retrieved. The key tools are class state encapsulation, method contracts (`push`, `pop`, `peek`), and defensive error signaling for invalid operations (for example, popping an empty stack).

In the control-tower arc, this class is the backbone of undo and recovery flows. If LIFO invariants fail, replay logic later becomes unreliable. The main tradeoff is API simplicity versus strictness: a permissive stack can hide bugs, while explicit exceptions surface mistakes early. Common misconception: a plain list is always enough. It can be the backing store, but the class boundary is what preserves stable semantics for the rest of the system. Reflection prompt: why is `peek` useful when no state should be removed yet?

## Assigned task(s)

- **Build/Change:** Implement `AuditStack` in `main.py` with explicit LIFO operations used by later lessons.
- **Expected outcomes / acceptance criteria:**
  - [ ] `push` adds entries so the most recent item is returned by both `peek` and the next `pop`.
  - [ ] Repeated `pop` calls follow strict LIFO order.
  - [ ] Empty-stack `pop` raises an appropriate exception (`IndexError`) instead of returning ambiguous sentinel values.
  - [ ] Stack length is externally observable (for example via `len(stack)`).
