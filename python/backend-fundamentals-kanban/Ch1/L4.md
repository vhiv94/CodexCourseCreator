# API Design Basics Methods Status and Errors

API design maturity begins where "endpoint exists" ends. By this stage, you already have a server, collection contracts, and nested resource modeling. Now you enforce consistency in method semantics, status code signaling, and error-envelope shape so clients can trust behavior across success and failure paths. In production systems, inconsistency in failure contracts costs more engineering time than many happy-path bugs, because every client integration now needs endpoint-specific defensive code.

This lesson centers on semantic discipline. Create operations should communicate creation semantics (`201` preferred, `200` acceptable if consistently justified). Missing resources should communicate not found (`404`) rather than generic client errors. Validation failures should use contract-correct client error statuses (`400` or `422`) and should be machine-parseable as JSON objects. Unsupported methods should surface `405` explicitly. Most importantly, different failure types should share at least one stable top-level key in the error payload (`detail` and/or `error` patterns), so clients can implement one reliable parsing strategy rather than brittle endpoint-by-endpoint exception branches.

The tools for this lesson are HTTP semantics, FastAPI exception handling patterns, and envelope design discipline. HTTP provides standardized status meaning; FastAPI provides ergonomic mechanisms for raising structured failures; JSON envelope conventions give clients a stable parsing surface. This is the final Ch1 project step because the conventions you set now become the baseline for all later backend evolution in Kanban Lite, including validation, integration tests, deployment diagnostics, and operational troubleshooting. Your objectives are to justify method/status choices from standards, ensure error payload serializability and shape consistency, and treat tests as formal behavioral contract evidence. Common misconceptions to reject: "status codes are optional if JSON explains enough," and "error shapes can vary freely because frontends can adapt."
