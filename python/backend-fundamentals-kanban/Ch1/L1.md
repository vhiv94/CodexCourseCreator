# What Is a Server

When backend engineers say "server," they are not naming a framework, and they are not naming a machine alone. In this course, a server is a long-running process that remains available, accepts requests over a network protocol, decides what behavior applies to each request, and returns a structured response. The process can run on your laptop, in a container, or on a VPS; the key idea is continuity of service and contract consistency, not location. Mechanically, this lesson asks you to establish a minimal but rigorous service boundary for Kanban Lite: the app must boot as a real FastAPI application surface, it must expose a deterministic health endpoint, and it must exhibit predictable behavior for unsupported or unknown requests.

The tools matter because each one solves a different boundary problem. FastAPI gives you an explicit request-routing surface and default HTTP semantics that can be reasoned about. HTTP itself gives you method meaning (`GET` vs `POST`) and status code semantics, which clients use as machine-readable evidence. JSON content types make payload expectations explicit for downstream consumers. The Chapter 1 project tie-in is direct: before you can model boards, columns, and tasks, you need a service shell that proves "there is a live backend process here, and its baseline contract is coherent." Your learning objectives in this lesson are to distinguish process-model thinking from framework-name thinking, to explain why liveness endpoints are read-only contract checks (not mutation surfaces), and to reason from tests as public specification. A common misconception is "if a route returns data once, the server concept is done." That is too weak. A backend contract includes method constraints, predictable response media type, and meaningful failure behavior on unknown paths.

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/health")
def health() -> dict[str, str]:
    return {"status": "ok"}
```

The snippet is intentionally tiny: it illustrates service shape and a liveness contract, not a full course solution.

Reflection prompts for your own reasoning: if a client sends `POST /health`, what architectural signal should your API give, and why is that signal useful to client maintainers? If an unknown path receives HTML in one place and JSON in another, what breaks first for API consumers: business logic, observability pipelines, or both?

## Assigned task(s)

- **Build/Change:** Establish a FastAPI application entry surface in `main.py` that resolves as `app` or `create_app()`, and define a read-only health contract at `GET /health`.
- **Expected outcomes / acceptance criteria:**
  - [ ] The application bootstraps to a real FastAPI instance that test clients can mount.
  - [ ] `GET /health` succeeds with HTTP `200` and a JSON object containing a stable `status` field (`ok`, `healthy`, or `up` semantics).
  - [ ] Health responses declare JSON content type, so monitoring/tooling does not infer payload format.
  - [ ] `POST /health` is rejected as method-not-allowed (`405`), demonstrating that liveness checks are not mutation endpoints.
  - [ ] Unknown routes return a not-found error (`404`) with a JSON error envelope shape suitable for API clients.
- **Evidence (tests):** `main_test.py` + `lesson_ch1_l1`
