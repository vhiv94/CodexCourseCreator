# Routing and Resource Modeling

Routing and resource modeling answer a foundational backend question: "What domain objects exist, who owns whom, and which URL patterns should encode that ownership?" In Kanban Lite, boards own columns, and columns own tasks in the board context. A route hierarchy is therefore not cosmetic naming. It is an executable boundary statement. When your API exposes `/boards/{board_id}/columns/{column_id}/tasks`, it tells both humans and machines that a task creation event is valid only under a coherent parent chain. This lesson is where backend design stops being generic HTTP mechanics and starts becoming domain architecture.

Mechanically, you are implementing nested resource surfaces with identity-aware behavior. Parent existence must be verifiable. Child existence must be contextual. Parent-child mismatch is a first-class failure mode, not an incidental bug. If a column belongs to board A, creating tasks for that column under board B should fail deterministically (`400`, `404`, or `409` are all defensible depending on your chosen error semantics). Unknown board and unknown column reads should communicate not-found behavior predictably. At the same time, this lesson introduces a persistence boundary signal: when successful write operations occur, SQLite-backed storage should initialize and be queryable. That does not mean "database internals are solved"; it means the request path now traverses into durable state rather than ephemeral memory alone.

Your tools in this lesson are path parameters, route composition, identifier propagation, and SQLite-backed write effects. FastAPI route declarations make ownership boundaries explicit at the path level. SQLite gives a durable persistence target and a stable queryable artifact after write operations. The project tie-in is pivotal: later chapters on SQL modeling, migrations, and reliability only make sense if your Ch1 routing model already encodes clear domain ownership. Your objectives are to reason about resource identity under nesting, defend parent-child mismatch handling as a contract requirement, and articulate where HTTP route correctness meets persistence initialization. Misconceptions to challenge directly: "nested routes are just URL aesthetics," and "if one create endpoint works, ownership constraints can be deferred."

```python
# Concept-only micro-shape for ownership, not assignment code
resource_path = "/boards/{board_id}/columns/{column_id}/tasks"
ownership_rule = "column_id must belong to board_id"
```

Reflection prompts: which failure is more dangerous for long-term data integrity, silently accepting mismatched parent-child identities or loudly rejecting them? Why is "SQLite file now exists after write" a useful boundary signal but not proof of complete schema correctness?
