# HTTP Request-Response Contract

An HTTP request-response contract is the shared behavioral agreement between client and server: which method is allowed on which path, what payload shape is accepted, what status family communicates each outcome, and what media type represents the response body. Without that agreement, even "working" APIs are operationally fragile because clients cannot distinguish valid outcomes from accidental behavior. In this lesson, you move from mere server liveness to contractual clarity for board resources: clients should be able to create a board, list boards, and predict how validation and method mismatch failures are expressed.

The mechanics are concrete. `POST /boards` consumes JSON input and yields a success status with JSON output for created resources. `GET /boards` returns a JSON collection shape, not an ad hoc object, because the route semantic is "list collection members." Validation boundaries matter as much as happy paths: missing required fields or wrong data types must fail with structured client-error statuses (`400` or `422`) so callers can recover intentionally. Method constraints also encode contract truth: when a route surface does not support `PATCH`, the server should communicate that explicitly with `405` rather than a vague alternative. This progression is why Ch1 places the lesson now: Kanban Lite is about resource workflows, and resource workflows collapse quickly when contracts are implicit.

FastAPI, request body modeling, and HTTP status semantics are the main tools in this lesson. FastAPI gives typed, explicit input boundaries; HTTP gives interoperable semantic categories; JSON and `Content-Type` communicate parse guarantees to any consumer language. Your objectives are to explain contract-first API behavior, justify status choices in terms of client actionability, and use tests as externalized evidence of contract obligations. Common misconceptions to resist: "a 200 response with any JSON is enough," and "method mismatch can be treated like not found." Those shortcuts reduce predictability and raise maintenance cost.

## Assigned task(s)

- **Build/Change:** Define contract-correct behavior for `/boards` collection endpoints covering creation, listing, validation errors, and unsupported methods.
- **Expected outcomes / acceptance criteria:**
  - [ ] `POST /boards` accepts JSON with a required `name` and returns success (`200` or `201`) with JSON that preserves the submitted board name.
  - [ ] Success responses for board creation declare JSON content type.
  - [ ] `GET /boards` returns `200` and a JSON array representing board collection state.
  - [ ] Missing required `name` fails with a structured client error (`400` or `422`) in a JSON error envelope.
  - [ ] Wrong `name` type (for example non-string input) fails with the same client-error semantics (`400` or `422`), preserving contract predictability.
  - [ ] Unsupported method usage on `/boards` (for example `PATCH`) is rejected with method-not-allowed semantics (`405`).
