# API Design Basics Methods Status and Errors

An API becomes much easier to trust when success and failure look consistent. Clients should not have to guess whether a missing board returns `404` today, `200` with a hidden error tomorrow, and a random text string next week. Good API design starts with repeatable semantics: successful reads usually return `200`, create operations should use a create-style success code such as `201`, missing resources use `404`, invalid input uses a validation-style failure such as `400` or `422`, and unsupported methods are rejected clearly with `405`. Just as important, error bodies should share a recognizable shape so both humans and client code know where to look for the explanation.

This lesson finishes Chapter 1 by turning scattered endpoint behavior into conventions. Those conventions are part of the capstone architecture, not just polish. Once you commit to stable method/status/error rules here, later lessons can add data access, validation, and deployment concerns without renegotiating basic client expectations every time. Helpful error payloads reduce frustration because they tell the client what category of failure happened and where to look next, instead of forcing guesswork from an empty or inconsistent response. Predictable APIs are easier to debug, easier to document, and much safer to evolve.
