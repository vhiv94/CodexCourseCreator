# Routing and Resource Modeling

Routes are more than URL strings. They are a public map of which resources your backend owns and how those resources relate to one another. In Kanban Lite, `/boards` represents the board collection, while `/boards/{board_id}/columns` represents the columns that belong to one specific board. Boards own columns, and columns own tasks, so the route structure should make that ownership visible instead of forcing the client to guess. Nested paths are useful here because they communicate containment and help the backend reject impossible combinations, such as trying to create a task inside a column that belongs to a different board. This lesson is also the first point where persistence becomes concrete: a successful write should initialize real durable state instead of disappearing after the request ends.

The big idea is not "make URLs longer." It is "make resource boundaries legible." A backend becomes easier to reason about when the path shape matches the domain model and when errors reflect parent-child truth instead of silently accepting mismatched ids. This is also where SQLite stops being abstract vocabulary: it is the local database engine storing the rows your successful writes create, so a board or column should still exist after the request finishes instead of living only in Python memory. That matters later when the API grows, because clear ownership boundaries make both your routing layer and your data layer less ambiguous.

- **Build/Change:** Model boards, columns, and tasks as related resources, and make successful write operations initialize SQLite-backed persistence.
- **Expected outcomes / acceptance criteria:**
  - [ ] A client can create a column under a specific board with `POST /boards/{board_id}/columns`.
  - [ ] A client can create a task under a specific board and column with `POST /boards/{board_id}/columns/{column_id}/tasks`.
  - [ ] Requests for unknown boards or unknown columns under a valid board are rejected with a not-found style response.
  - [ ] Parent-child mismatches are rejected instead of allowing a task to be created under the wrong board context.
  - [ ] At least one successful write operation initializes a SQLite-backed datastore that can be queried afterward.
