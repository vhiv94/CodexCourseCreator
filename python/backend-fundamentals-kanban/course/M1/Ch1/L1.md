# What Is a Server

A server is a program that keeps running so other programs can ask it for work. That sounds simple, but it changes how you think about code: instead of "run this script once," you are now designing a process that stays available, listens for requests, and answers them in a stable format. Here, "listen" does not mean vaguely waiting for a message from the internet. It means the process stays running, accepts HTTP requests on known routes, and turns them into responses a client can interpret. In this lesson, FastAPI is the tool that gives your backend a clear application surface, and HTTP is the protocol that lets clients ask, "Are you alive?" without needing to know your internal code. A health endpoint matters because it is the smallest honest proof that the service started successfully, can receive a request, and can return a machine-readable JSON response.

This lesson is intentionally narrow. You are not building the Kanban API yet. You are establishing the first public contract for it: there is a real FastAPI app, there is a route a client can call, and the service behaves predictably when a client uses the wrong method or asks for something that does not exist. That means `GET /health` should communicate service health, `POST /health` should be rejected as the wrong method with `405`, and unknown routes should fail clearly with a JSON `404`-style response instead of an HTML surprise. That foundation matters because every later board, column, and task endpoint assumes this application object and routing behavior already exist.
