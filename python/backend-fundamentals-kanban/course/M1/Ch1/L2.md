# HTTP Request-Response Contract

Once a server exists, the next question is what promises it makes to clients. An HTTP contract is that promise: which path a client should call, which method belongs there, what shape the request body should have, what content type comes back, and which status codes communicate success versus client mistakes. A request is not "asking for the whole backend." It is asking for one specific operation on one specific route, such as "list the boards" with `GET /boards` or "create a board from this JSON body" with `POST /boards`. FastAPI helps at the framework level, but the real design work is yours. You are deciding what counts as a valid board-creation request, what the API should send back after a successful write, and how the service should respond when the caller sends the wrong shape or the wrong method.

For Kanban Lite, this lesson turns a generic backend into the beginning of a useful API surface. The focus is still small: create a board and list boards. That narrow scope is helpful because it teaches the core HTTP ideas without hiding them inside too many features. JSON is the default here because it is easy for both Python code and future frontend clients to generate, parse, and validate without guessing at page structure. A good Chapter 1 contract is boring in the best way: clients know where to send JSON, what field is required, when a successful create should look like a creation response, and what kind of JSON they will get back when things go right or wrong.

## Assigned task(s)

- **Build/Change:** Define the initial `/boards` request-response contract so clients can create boards and list existing boards with predictable HTTP behavior.
- **Expected outcomes / acceptance criteria:**
  - [ ] `POST /boards` accepts a JSON body with a string `name` and returns a successful JSON response for the created board.
  - [ ] The successful create response preserves the board name and uses a JSON content type.
  - [ ] `GET /boards` returns `200` with a JSON array rather than an HTML page or plain-text dump.
  - [ ] Missing `name` values or clearly wrong types are rejected with a validation-style status such as `400` or `422`.
  - [ ] Unsupported methods on `/boards`, such as `PATCH`, are rejected with `405`.
