# L3: Comment What Your Script Does

## Concept

A **comment** is text in your source file that **Python does not execute**. Full-line comments start with `#`; everything after `#` on that line is for people reading the script. They do not change what runs, but they answer questions like *why is this here?* and *what is this program heading toward?*

On tiny scripts it is still worth commenting the **intent**: what the visible output is for, and how this file fits into a larger goal. That habit scales when the program grows—you are documenting **purpose**, not narrating every keystroke.

You already built a **multi-line banner** (L2). This lesson keeps that behavior and adds **at least two** meaningful full-line comments: one should speak to the **banner or printed output**, and another to the **study-log direction** the course is building, even though logging is not implemented yet.

```python
# Example only: Python skips this when running; readers see the explanation.
```

## Why it matters in this project

The study-log CLI will get more moving parts—input, validation, files, modes. Comments that anchor *what this block is for* save future-you (and reviewers) from reverse-engineering intent from output alone. The spine’s `project_step` asks for comments that connect the **banner** you print now to the **study log** you are growing toward.

## Common misconceptions

- **“Comments can be super short as long as there are two.”** — The tests require each full-line `#` body to be at least **eight** characters and to cover **banner** and **study/log** themes across the set.
- **“Inline comments next to code count.”** — Only lines where, after stripping, the line **starts with `#`** are collected. Put explanations on their own `#` lines if you want them picked up reliably.
- **“I can change the banner to one line again.”** — L3 still enforces the **multi-line** banner rules from L2.
