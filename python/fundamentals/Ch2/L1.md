# L1: Remember Values with Variables

## Concept

A **variable** is a **name** you choose in source code; with **assignment** (`=`) you bind that name to a **value** so you can reuse it, read it aloud in your head, and change it in one place later. For strings, the value often comes from a **string literal** in quotes.

Naming is part of the design: `user_age = "twelve"` is legal Python, but a misleading name hurts readers. This lesson focuses on **two readable string “constants”** your study-log program will keep coming back to: something that represents the **app’s visible title** (what users see in the banner area) and something that represents the **default log file** on disk. They should live in **different** variables so “what we show” and “what file we mean” never get confused.

```python
# Illustration only: a name bound to a string literal (not your full program).
greeting = "Hello"
```

## Why it matters in this project

Chapters ahead add input, loops, and file I/O. If the banner text and log path stay **magic strings** scattered through `print` calls and open-save logic, small renames become error hunts. Pulling them into **well-named variables** now matches the spine’s direction—**title** and **default log filename**—and prepares you for configuration-style thinking later in the course.

## Common misconceptions

- **“Any two strings count.”** — The tests distinguish **title-like** names/values from **filename-like** names/values using patterns in `Ch2/L1.py`. A “title” stored in a variable whose **value** looks like `notes.txt` will not satisfy the title side.
- **“One variable can hold both ideas.”** — You need **two** different variable **names**; reusing the same identifier for both roles fails the disjoint check.
- **“Fancy assignments count.”** — Only **simple** `name = "literal"` (and annotated forms the visitor handles) are collected. Building strings only with `+` or `print(f"...")` without the literal assignments the test looks for will not establish the constants the lesson targets.
