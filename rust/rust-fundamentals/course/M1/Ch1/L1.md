# Create and run the Log Summary crate

**Cited contract:** [`course/spine.yaml`](../../spine.yaml) (M1 → Ch1 → L1), [`course/overview.md`](../../overview.md) (Log Summary CLI arc).

Cargo is the everyday front door for Rust on your machine: it creates projects, resolves dependencies, compiles, and runs binaries with predictable commands. For this course, every lesson extends one long-lived crate—the **Log Summary CLI**—so the first win is simply proving that toolchain and project skeleton work on *your* system. A binary crate is the right shape here because the capstone is a terminal program, not a library reused by other Rust code (library crates are useful too; this course just does not start there).

The normal workflow you will internalize is: create the crate once, then iterate with `cargo run` while you edit Rust sources under `src/`. `cargo build` compiles without executing; `cargo run` compiles if needed and then runs the default binary. You do not need custom scripts for this lesson. One misconception is that you must invoke `rustc` by hand—Cargo calls the compiler for you and records how the crate is wired in metadata files you will tour in the next lesson.

This lesson stays intentionally small: you only establish the crate and run the starter program. Formatting output, reading stdin, and richer CLI behavior come in Ch1/L2–L4.

## Assigned task(s)

- **Build/Change:** With Cargo, create a **new binary crate** that will host the Log Summary CLI for the whole course. The usual starting command shape is `cargo new --bin <crate-name>` from the parent directory of where you want the folder. Pick a package name you are happy to keep on disk (for example `log-summary-cli` in kebab-case). Put the crate where you plan to keep learner work—a fresh folder is fine and sidesteps any unrelated VCS experiments.
- **Expected outcomes / acceptance criteria:**
  - [ ] Running the default binary via Cargo prints the starter `Hello, world!` message (or the small default output Cargo generates for a binary) without errors.
  - [ ] You can name at least the two most important paths Cargo creates for this layout: the manifest at the crate root and the primary source file under `src/`.
  - [ ] You use Cargo’s normal run path (for example `cargo run` from the crate root) rather than compiling with ad hoc `rustc` flags for this step.
- **Evidence:** From the crate root, capture or note the successful `cargo run` output and, in your own words, what each of those two paths is for (one sentence each is enough).

**Optional — debugger-first habit:** If you prefer stepping through code over printf debugging, install the **CodeLLDB** extension in VS Code or Cursor, open the crate folder, and try “Run and Debug” on the generated binary target once `cargo build` succeeds. Rust Analyzer (the `rust-analyzer` extension) pairs well with that workflow. You do not need perfect launch configuration on day one; the goal is to confirm you can hit `fn main()` and step once.
