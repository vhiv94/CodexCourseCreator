# Checkpoint on first Cargo and CLI input steps

**Cited contract:** [`course/spine.yaml`](../../spine.yaml) (M1 -> Ch1 -> L5), [`course/overview.md`](../../overview.md) (Log Summary CLI arc). **Builds on:** [L4 - Capture one line of user input](L4.md).

This lesson is a checkpoint, not a new implementation push. By the end of Chapter 1, you have already created a Rust binary crate, located the manifest and entry point, printed a small structured welcome, and read one line from stdin into a `String`. Before moving into variables and control flow, pause long enough to make sure that workflow makes sense as a connected loop rather than four isolated facts.

The goal here is explanation, not memorization. If you can point at your current crate and describe what `Cargo.toml` is for, where `fn main()` fits, what `println!` bought you, and how `read_line()` changed the program from output-only to interactive, that is enough. A short written answer or tiny demo transcript is a valid completion artifact.

## Assigned task(s)

- **Explain/Review:** Answer a short checkpoint in your own words using your current Log Summary CLI as the reference point.
- **Expected outcomes / acceptance criteria:**
  - [ ] You can explain the difference between `Cargo.toml` and `src/main.rs`.
  - [ ] You can describe the role of `fn main()` when the program starts.
  - [ ] You can explain how your current CLI prints output and captures one line of input.
- **Evidence:** A short-answer note, chat response, or tiny demo transcript is enough.

Suggested prompt set:

1. What is the difference between `Cargo.toml` and `src/main.rs` in your crate?
  - main.rs is where I write source code, and cargo.toml is where I keep project architecture information.
2. What role does `fn main()` play when you run the program?
  - fn main() is the entry point that the os points to once it creates a process for the program.
3. What is `println!` doing for you in the current CLI?
  - it formats the string inputs (I can pass variables and even interperate data in different formats, i.e. hexadecimal or binary)
4. How did you capture one line of user input from stdin, and what data type did you store it in?
  - first I had to create a `String` buffer to hold the result. it differs from str. Then I created a handle to stdin `Stdin()` and called it's method `read_line(&mut buf)` passing the buffer to receive the input, as well as locking the stdin buffer. the method stops at the first `\n`, hence read_**line**.
5. Why can `read_line()` make echoed output look like it has an extra blank line?
  - because read_line captures the `\n` that it's waiting for. to get rid of that simple call `.trim()` on the String buffer after to trim any leading or trailing whitespace.
6. If you wanted to pause until the user presses Enter, why would `read_line()` usually be the more natural first choice than `read_exact()` on a normal terminal?
  - read_line because read exact will be reading until EOF
