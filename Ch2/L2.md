# L2: Read a Line from the Keyboard

## Concept

Programs become interactive when they **wait for you to type** and then **read what you typed**. In Python, **`input(...)`** pauses execution, shows an optional **prompt** string in the terminal, and returns what the user entered as a **string**—usually including the newline-ending behavior of the console, which you routinely normalize.

People often add **accidental spaces** before or after the words they care about. For a study note, you usually want the **meaningful text** only. String method **`.strip()`** removes leading and trailing whitespace so `"   hello   "` becomes `"hello"`. Combine **one** careful `input` with **strip** before you **display or store** the line.

```python
# Illustration only: prompt + strip pattern (not your full program).
raw = input("Type something: ")
clean = raw.strip()
```

## Why it matters in this project

The study log is built from **what the user types**. If you echo padded garbage or skip stripping, every later step—validation, saving, searching—gets harder. This lesson adds a **single** “read one note line” step: prompt clearly, read once, **strip**, then surface the cleaned text so you can see the habit before loops and `if` checks arrive.

## Common misconceptions

- **“I'll call input() in a loop for safety.”** — This lesson’s tests expect **exactly one** `input(...)` in `main.py`. Save multi-line or repeat reads for later lessons.
- **“Strip doesn’t matter if the user types carefully.”** — The test **always** sends padding; output must reflect **stripped** text and avoid echoing the padded line whole.
- **`input()` with no prompt** — The tests require something inside `input(...)` so the prompt is visible (positional string or `prompt=...`).
